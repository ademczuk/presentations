The one-line version
AgentReef is DNS + RPC + IAM for a fleet of AI agents

DNS

Find a capability by name instead of memorizing where it lives.

RPC

Call a function on somebody else's machine and get a signed answer back.

IAM

Identity and access management: who is allowed to do what.

Three boring old ideas. They're boring for a reason. They work.
Two words I'll lean on
A node is a signed peer identity. An owner is the human behind it.
Node

A running agent, its own key

Not a person. A cryptographic identity that can prove it is the thing it claims to be. My GPU box is a node. My laptop is a node. Different keys, same owner.

Owner

The human or org behind it

Every message on the mesh is signed by a node. Every signature checks against an owner. If it doesn't check out, it goes in the trash.

Your key never leaves your machine. Not once. Not to me, not to the relay, not to anybody.
Architecture
Control plane vs data plane
Durable governance splits from runtime traffic. Control plane: auth, registry, policy, audit. Data plane: routing, shared memory, namespace-scoped context.

The trust part

Signed result back
If the relay can't execute, the relay can't lie. Every call and every result is Ed25519-signed. A compromised relay cannot forge a result.

One door, and it only opens outbound

Each client machine has one daemon/router boundary. Remote work crosses through your own router, never raw node to node. Being on the mesh adds zero inbound exposure to your box.

The exact mechanism
Twelve steps. Nothing in here is "trust me."
Signed call in, authorized against the grant, quota-reserved, admitted under a Postgres lock, routed by presence, invoked, and the signed result delivered back.
What's actually in the row
One row. One column you flip. One command to undo it.

The base row

  • grantor and grantee owner
  • the specific peer granting
  • skill name + a version pin
  • expires_at
  • revoked_at, starts empty

Stacked on top

  • manifest hash (rug-pull defense)
  • optional human-consent gate
  • optional offline capability token
  • execution-count cap
  • metered: try_reserve calls_per_day
Revoke stamps revoked_at. Next call checks the row, sees the stamp, dies. No key to rotate, no secret to change.

Nothing shared unless you say so
Postgres row-level security. The app can't bypass it either.

NOTSUPERUSER / NOBYPASSRLS

Not "the code checks the tenant." The database itself refuses the row if you're not in it. A bug in my code still can't leak your data across a tenant boundary.

Three deliberate acts to share anything: put the skill on the dev plane, into a trust group, and write a grant.

Propose-only by construction
Four walls. Four different keyholders.
The model does not decide to behave. Every layer assumes it is already broken.

1. The grant

Write-only to a proposals folder. Ask it to touch the wiki and the server says no. Not the model. The server. Live-tested: the wiki write came back permission denied.

2. The card

Every proposal hardcodes one verdict: propose. There is no apply field to serialize. You physically cannot write the instruction to merge.

3. The validator

A different identity rejects any bad card, and gemma cannot read that reject log. It never sees its own rejections, so it cannot learn to game them.

4. Liveness

An off-host check attests it is actually alive and actually gemma, before anything it says counts.

200 lies
A health check that says 200 is lying to you.
Measured on this box: /health answered 200 in 0.24 seconds while real completions hung for 48 seconds, the model still resident at 21 GB, 5 percent utilization.

No echo: exit 2. Wrong model: exit 3. Green: exit 0. Fail closed.
4
Run it off-host, as an identity that is not gemma
The node cannot vouch for itself.
A signature proves the key, not the intent. Gemma's key has one holder, and everything that judges gemma runs as something that isn't gemma.

Q&A appendix · cadence and guards
The unattended schedule

Audit lanes

  • Backlog deconfliction: every 30 minutes
  • Knowledge graph curation: every 2 hours
  • Hardening triage: daily, out-only always
  • Kill switch: one file on disk pauses every lane
  • Single-pass lock: one audit at a time, the GPU is shared

Watchdogs (two, different jobs)

  • Canary watchdog: probe every 120s, 3 non-greens = down, one green clears
  • Latency early-warning at 15s: pressure flag before a wedge
  • Server watchdog: real-completion probe, 3 strikes, max 3 restarts per hour
  • Presence beat reads "active" only off a green canary, never a bare heartbeat

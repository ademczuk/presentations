Own Your AI #2 · 20-minute talk
AgentReef:
Verified Loops for Coding Agents
Autonomy is cheap now. Trust is the bottleneck.
Tickets become loops. Acceptance criteria become the stop condition. One agent implements, another verifies, and a human holds the merge button.
Andrew Demczuk · builds AgentReef (22-crate Rust mesh) · merges agent-written code daily
I ran one query against my own repo
24 / 40
of my last 40 merged pull requests had zero reviews
Nobody looked. Not a human, not a bot. I merged them because the diff looked right and it was late.
The code was mostly fine. That's what makes it dangerous. Mostly-fine merges are how you build a codebase where nobody remembers which corners are load-bearing.
Agents made this worse: they produce more PRs in a night than I used to write in a month. Generation scaled. Trust did not.

Four of these five are stock tooling you can steal tonight. The fifth is why I built a mesh.
Move 1
The ticket is the contract
What

573 task files in the backlog. 550 carry a numbered Acceptance Criteria section, wrapped in machine-readable markers, with a CLI to tick each one.

Why

If the done-condition is not written down, the agent decides what done means. Its favorite definition: "I produced text that resembles done."

How

backlog task edit TASK-561 --check-ac 4

PRs cite criteria by number. A real merged title: "TASK-561 AC4: reply-row projection and ack identity."

A ticket without acceptance criteria is a wish. "Does this diff satisfy AC4" is answerable. "Is this good code" is not.
Move 2
The ticket becomes a loop
$ /ralph-loop "implement TASK-561 AC4" --max-iterations 12 --completion-promise "AC4 CHECKED AND TESTS GREEN"
# a Stop hook intercepts every session stop:
# promise met (literal string match, inside promise tags) = loop ends
# promise not met = the SAME prompt is fed back in. again. and again.
Why a loop

A single-shot agent stops at looks-done. A loop with a promise stops at is-done. The gap between those two is every bug you ever shipped at midnight.

The number to respect

Default max iterations is 0 = unlimited. No promise, no cap: an agent that runs until your subscription taps out. Set both. Every time.

The promise check is literal string equality. Character for character. A completion handshake, not a vibe.
Move 3
The implementer cannot push
What

Every implementing agent runs in its own git worktree, on its own branch named after its task, as its own session. Coordination is mail, not a shared checkout.

The enforcement

The builder's git push is not discouraged. It is intercepted: a hook in the worktree settings blocks the tool call before it runs. Not a polite sentence in the prompt.

At fleet scale

Overnight fleets: commit per ticket, ticket id leading the message, a progress file as the done signal. Fleets never push. The operator pushes.

An agent that can push can also push a fix to the test that would have caught it.
Move 4 · the heart
A different agent verifies. Two lines of config.
# .github/CODEOWNERS
* @AnisminC # every PR routes to the reviewer agent
/.github/CODEOWNERS @ablac # the gate itself belongs to the human
The one file the reviewer does not control is the file that appoints the reviewer. It cannot rewrite its own gate.

Draft-first, and the ping is instant

Draft PR: silence, no tokens burned. Mark it ready: the review request fires to the agent in under one second (measured on three consecutive PR timelines). Draft while you cook. Ready when you mean it.

Why a DIFFERENT agent: the implementer's context is full of its own justifications. The reasoning that produced the bug will happily approve the bug. You need a colder reviewer, not a smarter one.

An approval that survives new commits is a rubber stamp with a timestamp.
Arena repo, PR #48, spectator wounded-state feature. Merged after this cycle.
Move 4 · it says no
It blocked my own PR this month
CHANGES_REQUESTED on PR #169, pinned to commit 27c7382

"Reviewed PR #169 at 27c7382d. The policy-registry design is
structurally sound. Blocking: the PR is not rustfmt-clean on
changed Rust files."
I built this thing, and it bounced my pull request over formatting. Good. That is the system working.
A verifier that never rejects anything is not a verifier.
Move 5
A human holds the merge button
# repo instructions, line 5, verbatim:
NEVER push to main directly. Ever.
main is updated only through a reviewed PR
# enforced by a pre-push hook, not by discipline
The biggest threat to this process was never the agents. The 24 unreviewed merges were all mine. The hook exists because of me.
What happens if you skip this move? That's my lightning talk tonight. Short version: the robot approves itself, everything goes green, nothing ships.
The fifth move needs infrastructure
"Different agent" should be cryptographic, not cosmetic

A node is a key, not a prompt

Every AgentReef agent holds its own Ed25519 signing key. The reviewer is not a reviewer-flavored system prompt. It is a different identity, and every call it makes is signed.

The war story

Two nodes once ran off the same key file. Real signatures, cryptographically valid, wrong actor. A signature proves the key, not the intent. One key, one holder.

Everything that judges an agent runs as an identity that is not that agent.
Permission is a row in Postgres
Grants: scoped, pinned, capped, revocable

The row

  • grantor, grantee, peer
  • skill name + version pin
  • expires_at
  • revoked_at: stamp it, the next call dies at the receiver

Execution cap

"You get 50 calls," not calls forever. A compromised agent at 100 ops per tick fits ~6,000 unauthorized calls inside a 60-second revocation window. The cap bounds damage regardless of agent speed.

Manifest hash

The skill's behavior-defining manifest, hashed at grant time, recomputed on every invoke. Skill quietly changes what it does: hash breaks, call dies. Rug-pull closed.

Underneath: a 10-state call machine where every transition is guarded in the SQL itself. The database is the arbiter. Two racing processes cannot both complete the same call.
The mechanism, end to end
Twelve steps. None of them is "trust me."
Signed frame in, authorized against the grant, quota reserved, admitted under a lock, routed by live presence, invoked on the target, signed result delivered back.
You prove a fence by pushing on it
The demo that matters is the deny
Granted

shared/curation-proposals

My auditor node holds memory.write scoped to exactly this one namespace. Its whole write surface.

Denied, live-tested

shared/wiki

Never granted. The attempted write came back permission denied, at the receiver, before anything happened.

And a guard watches the substrate itself

Shared skills are treated as supply-chain artifacts, shared memory as untrusted input. Verdicts: approved, quarantined, blocked. A misalignment auditor walks the signed tool-call traces for untrusted sources reaching sensitive sinks. Honest caveat: that auditor is advisory today. It flags into a review queue. Flagging is not enforcement, and I know the difference.

What it costs
Match the ceremony to the blast radius
Change classCeremony
Docs, typoSelf-review. Ship it.
Single-file fixRun it as a loop with tests.
Multi-file featureLoop plus a second-agent verify.
Schema, auth, anything concurrentLoop, verifier, and a human actually reading the diff.

The bill

Latency and tokens. A verified loop is at least two agents where one used to be, plus review round-trips. Verification is not free. Rollbacks at 3am are less free.

And the confession

The review gate on my repos is 19 days old at this event. The 24 zero-review merges all predate it. I am not claiming I always worked this way. The gate exists because I didn't.

The pattern in one page
Steal these five lines
1
Write acceptance criteria a machine can check
The ticket is the contract.
2
Run the work as a loop whose stop condition is that contract
Cap the iterations. Always.
3
Implementer in a worktree, push mechanically blocked
Read-only remote by construction, not by request.
4
Verifier on a different identity, approvals pinned to commits
Fresh context. New commits dismiss old approvals.
5
A human merges
Always.
Own the weights. Own the box. Own the merge.
